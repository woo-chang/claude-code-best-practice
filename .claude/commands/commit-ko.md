# 한글 커밋 커맨드

변경사항을 분석하고, `commit-rules` 스킬의 7가지 검증을 실행한 후, 한글 커밋 메시지를 생성하여 커밋합니다.

## 워크플로우

### 1단계: 변경사항 분석

다음 git 명령을 **병렬로** 실행합니다:

1. `git status` — 변경/추가/삭제된 파일 확인
2. `git diff --staged` — 이미 staged된 변경 내용 확인
3. `git diff` — unstaged 변경 내용 확인
4. `git log --oneline -5` — 최근 커밋 메시지 참조
5. `git branch --show-current` — 현재 브랜치 확인

### 2단계: 커밋 전 검증 (7가지)

`commit-rules` 스킬의 "파트 2: 커밋 전 검증 체크리스트"를 순서대로 실행합니다.

#### 검증 1: 민감 파일 감지
`git diff --staged --name-only`에서 다음 패턴을 검색합니다:
- `.env`, `.env.*`
- `credentials`, `secret`, `token`이 포함된 파일명
- `*.pem`, `*.key`, `*.p12`
- `*password*`, `*apikey*`
- **발견 시**: 🚫 차단 — "민감 파일이 staging에 포함되어 있습니다: [파일명]"

#### 검증 2: 브랜치 보호
`git branch --show-current`가 `main` 또는 `master`이면:
- **발견 시**: 🚫 차단 — "main 브랜치에 직접 커밋할 수 없습니다. `ko` 브랜치에서 작업하세요."

#### 검증 3: 프론트매터 정합성
staging된 `.md` 파일을 Read로 읽어 프론트매터 영역을 검사합니다:
- `---` 마커 사이에서 `:`  앞의 키가 영어인지 확인
- **한글 키 발견 시**: ⚠️ 경고 — "[파일명]의 프론트매터 키에 한글이 포함되어 있습니다: [키명]"

#### 검증 4: 코드 펜스 내 한글 감지
staging된 파일에서 코드 펜스(` ``` `) 내부의 한글 문자를 검색합니다:
- 코드 펜스 열기/닫기를 추적하면서 내부 영역의 한글(가-힣)을 탐지
- **발견 시**: ⚠️ 경고 — "[파일명] 줄 [N]: 코드 펜스 내부에 한글이 있습니다: [내용]"

#### 검증 5: 내부 링크 유효성
staging된 `.md` 파일에서 `](상대경로)` 패턴을 찾아 대상 파일 존재를 확인합니다:
- 외부 URL (http://, https://)은 건너뜁니다
- 프래그먼트 앵커(#)만 있는 링크는 건너뜁니다
- **대상 파일 없음**: ⚠️ 경고 — "[파일명]의 링크 대상이 존재하지 않습니다: [경로]"

#### 검증 6: JSON 유효성
staging된 `.json` 파일을 `python3 -m json.tool`로 검증합니다:
- **파싱 실패 시**: 🚫 차단 — "[파일명]의 JSON 구문이 유효하지 않습니다: [오류]"

#### 검증 7: 화이트리스트 외 파일 수정 감지
staging된 파일의 확장자를 확인합니다:
- `.py`, `.svg`, `.png`, `.jpg`, `.gif`, `.webp`, `.wav`, `.mp3` 확장자
- 파일명이 `output.md` 또는 `MEMORY.md`
- **발견 시**: ⚠️ 경고 — "번역 대상이 아닌 파일이 수정되었습니다: [파일명]"

#### 검증 결과 처리
- 🚫 **차단** 항목이 하나라도 있으면: 커밋을 중단하고 문제 목록을 표시합니다. 3단계로 진행하지 않습니다.
- ⚠️ **경고** 항목만 있으면: 경고 목록을 표시하고 AskUserQuestion으로 "경고 항목이 있습니다. 계속 진행할까요?" 확인을 받습니다. 사용자가 취소하면 중단합니다.
- ✅ **모두 통과**: 3단계로 진행합니다.

### 3단계: 변경 분류

분석 결과를 바탕으로 분류합니다:
- **유형**: feat/translate/update/remove/fix/config/verify 중 적절한 영어 접두사 선택
- **범위**: 변경된 파일 수와 디렉토리
- **내용**: 무엇이 어떻게 변경되었는지 요약

### 4단계: 커밋 메시지 생성

`commit-rules` 스킬의 형식에 따라 커밋 메시지를 작성합니다:
- 요약 줄: `<영어 접두사>: <한글 변경 요약>` (70자 이내)
- 파일이 3개 이상이면 본문에 목록 포함
- `Co-Authored-By: Claude <noreply@anthropic.com>` 포함

### 5단계: 사용자 확인

AskUserQuestion 도구로 다음을 표시합니다:
- 검증 결과 요약 (✅ 통과 / ⚠️ 경고 수)
- 생성된 커밋 메시지 전문
- "이 메시지로 커밋할까요?"
- 선택지: "확인", "메시지 수정", "취소"

### 6단계: 커밋 실행

사용자가 확인하면:
1. 변경된 파일을 `git add`로 staging (특정 파일명으로 추가, `git add -A` 사용 금지)
2. HEREDOC 방식으로 커밋:
```bash
git commit -m "$(cat <<'EOF'
<커밋 메시지>

Co-Authored-By: Claude <noreply@anthropic.com>
EOF
)"
```
3. `git status`로 커밋 성공 확인

## 핵심 요구사항

1. **검증 먼저**: 커밋 메시지 생성 전에 반드시 7가지 검증을 먼저 실행하세요
2. **차단은 절대적**: 🚫 차단 항목이 있으면 어떤 상황에서도 커밋하지 마세요
3. **경고는 사용자 판단**: ⚠️ 경고는 사용자에게 보여주고 판단을 맡기세요
4. **commit-rules 스킬 준수**: 유형 접두사, 메시지 형식, attribution을 반드시 따르세요
5. **사용자 확인 필수**: 커밋 전 반드시 AskUserQuestion으로 확인을 받으세요
6. **특정 파일 staging**: `git add -A`나 `git add .` 대신 파일명을 명시하세요
7. **amend 금지**: 항상 새 커밋을 생성하세요
